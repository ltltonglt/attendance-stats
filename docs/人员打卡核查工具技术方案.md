# 人员打卡核查工具技术方案

## 1. 项目概述

本工具用于根据人员信息表和多个打卡数据导出文件，自动核查人员签到、签退、8 小时工时、打卡制式、用车车牌一致性，并输出新的 Excel 核查结果表。

工具提供桌面 GUI，用户可选择：

- 人员信息表；
- 打卡数据导出文件夹；
- 核查结果输出文件；
- 是否同时合并打卡数据。

当前实现语言为 Python，Excel 读写使用 `openpyxl`，桌面界面使用 Python 标准库 `tkinter`。

## 3. 输入与输出

### 3.1 人员信息表

默认读取第一个 Sheet。

关键列固定为：

| 列 | 字段 | 用途 |
| --- | --- | --- |
| B | 姓名 | 主匹配字段 |
| C | 用户邮箱 | 重名人员辅助匹配字段 |
| D | 打卡制式 | 与打卡数据 `主产品` 做一致性判断 |

输出时保留人员信息表 A-F 列。

### 3.2 打卡数据导出文件夹

读取文件夹下所有非临时 `.xlsx` 文件，忽略 `~$` 开头的 Excel 临时文件。

默认读取每个打卡文件的第一个 Sheet。

必要字段：

- `考勤类型`
- `姓名`
- `用户邮箱`
- `考勤时间`
- `站点编码`
- `主产品`
- `用车场景`
- `车牌号`

签到、签退、工时、制式和车牌核查只使用 `考勤类型 = 签到` 的记录。`请假`、`出差` 等记录用于站点核查的日期范围例外判断。

### 3.3 核查结果表

输出表保留 A-F 原始人员信息，并新增 G-P 列：

| 列 | 字段 |
| --- | --- |
| G | 打卡项目是否符合 |
| H | 打卡站点核查 |
| I | 签到时间 |
| J | 签退时间 |
| K | 是否满足8小时 |
| L | 打卡制式是否一致 |
| M | 签到车牌 |
| N | 签退车牌 |
| O | 车牌是否一致 |
| P | 备注 |

签到车牌、签退车牌正文单元格不留空，无值时输出 `/`。

## 4. 核心匹配规则

### 4.1 人员匹配

优先按姓名匹配。

如果人员信息表中存在重名人员，则该姓名对应的人员使用 `姓名 + 用户邮箱` 匹配打卡数据。

示例：

- 非重名：`姓名`
- 重名：`姓名 + 用户邮箱`

### 4.2 打卡项目核查

项目核查输出到 `打卡项目是否符合` 列，结果包括：

- `符合`
- `不符`

基础规则：

- 打卡数据文件名去掉 `.xlsx` 后即为该文件的项目号；
- 人员信息表 A 列 `项目号` 与打卡数据文件名项目号一致时，判定为 `符合`；
- 如果员工任一签到记录来自不一致的项目文件，则判定为 `不符`；
- 没有签到记录时判定为 `不符`。

项目号归一规则：

- 人员信息表项目号末尾括号备注会被忽略；
- 例如 `2023年四川移动技术服务项目(2人3车）` 归一为 `2023年四川移动技术服务项目`。

### 4.3 打卡站点核查

站点核查输出到 `打卡站点核查` 列，结果包括：

- `符合`
- `不符`

基础规则：

- 员工 `考勤类型 = 签到` 的打卡记录中，`站点编码` 必须包含 `2025年新`；
- 只要存在一条签到记录的 `站点编码` 不包含 `2025年新`，默认判定为 `不符`。

例外规则：

- 如果员工存在 `考勤类型 = 请假` 或 `考勤类型 = 出差` 的记录；
- 且该请假/出差记录的 `考勤时间` 日期范围覆盖了该员工签到记录的日期；
- 则该签到记录即使站点编码不包含 `2025年新`，也按符合处理。

员工所有签到记录均满足基础规则或例外规则时，整体判定为 `符合`。没有签到记录时判定为 `不符`。

### 4.4 签到与签退时间

第一优先级：

- 仅使用 `用车场景 = 无` 的签到记录；
- 最早 `考勤时间` 作为签到时间；
- 最晚 `考勤时间` 作为签退时间。

补充规则：

- 如果某员工 `用车场景 = 无` 的记录少于 2 条，则将 `开始用车` 和 `结束用车` 记录也纳入候选；
- 在候选记录中仍按最早 `考勤时间` 作为签到时间，最晚 `考勤时间` 作为签退时间。

如果选中的时间来自用车记录，则在备注列填写：

- `签到时间使用开始用车时间`
- `签退时间使用结束用车时间`
- 两者都发生时：`签到时间使用开始用车时间，签退时间使用结束用车时间`

### 4.5 是否满足 8 小时

当签到时间和签退时间都存在，且：

```text
签退时间 - 签到时间 >= 8 小时
```

则填写 `是`，否则填写 `否`。

### 4.6 打卡制式一致性

使用选中的签到时间、签退时间对应记录的 `主产品` 与人员信息表中的 `打卡制式` 对比。

当前按关键词判断：

- 人员表 `打卡制式` 包含 `5G NR` 或 `5GNR`，则匹配打卡数据 `主产品` 中的 `5G NR` 或 `5GNR`；
- 人员表 `打卡制式` 包含 `LTE`，则匹配打卡数据 `主产品` 中的 `LTE`；
- `专家` 等前缀不影响判断。

输出结果：

- `一致`
- `签到制式不一致`
- `签退制式不一致`
- `签到和签退制式不一致`

### 4.7 车牌一致性

车牌判断基于 `开始用车` 与 `结束用车` 记录。

基础规则：

- 没有开始用车和结束用车记录：`未用车`
- 缺少开始用车：`缺少签到`
- 缺少结束用车：`缺少签退`
- 开始车牌与结束车牌一致：`一致`
- 开始车牌与结束车牌不一致：`不一致`

多条用车记录规则：

- 如果存在多条 `开始用车` 或多条 `结束用车`，会在开始/结束车牌集合中查找是否存在相同车牌；
- 只要存在任意一组相同车牌，即判定 `一致`；
- 输出的签到车牌和签退车牌使用匹配成功的车牌；
- 如果没有任何匹配车牌，再按默认规则取开始用车记录和结束用车记录进行不一致判断。

备注补充：

- 多条开始用车：`存在两条开始用车`、`存在三条开始用车` 等；
- 多条结束用车：`存在两条结束用车`、`存在三条结束用车` 等。

## 5. 合并打卡数据功能

GUI 中提供复选框：

```text
同时合并打卡数据
```

默认不勾选。

勾选后，工具会额外输出合并后的打卡数据文件：

```text
<核查结果文件名>_合并打卡数据.xlsx
```

合并规则：

- 合并打卡数据导出文件夹下所有非临时 `.xlsx` 文件；
- 第一列新增 `项目号`，值为打卡文件名去掉 `.xlsx` 后的项目名；
- 保留原始打卡导出表所有列，并整体右移；
- 最后一列新增 `来源文件`；
- 合并功能保留所有原始记录，包括 `签到`、`请假`、`出差` 等；
- 如果不同打卡文件表头不一致，则终止合并并提示错误。

## 6. GUI 设计

GUI 入口文件为：

```text
attendance_audit_gui.py
```

主要控件：

- 人员信息表选择框；
- 打卡数据文件夹选择框；
- 输出文件选择框；
- `同时合并打卡数据` 复选框；
- `开始核查` 按钮；
- 日志输出框；
- 状态栏。

处理逻辑在后台线程中运行，避免 GUI 卡死。

GUI 启动时会自动设置 Tcl/Tk 路径兜底：

- `TCL_LIBRARY`
- `TK_LIBRARY`

用于减少不同环境下 Tk 初始化失败的问题。

## 7. 异常处理

核心异常类型为：

```python
AttendanceAuditError
```

主要异常场景：

- 人员信息表不存在；
- 打卡数据文件夹不存在；
- 打卡数据文件夹中没有可读取的 `.xlsx`；
- 打卡文件缺少必要字段；
- 考勤时间无法解析；
- 输出文件被 Excel 占用导致无法保存；
- 合并打卡数据时表头不一致。

GUI 捕获异常后会在日志区和弹窗中提示。

## 10. 运行方式

在项目目录执行：

```powershell
python attendance_audit_gui.py
```

也可在 Python 代码中直接调用核心函数：

```python
from attendance_audit_core import audit_personnel

audit_personnel(
    "人员信息.xlsx",
    "打卡数据导出",
    "人员打卡核查结果.xlsx",
    merge_attendance=False,
)
```

如果需要合并打卡数据：

```python
audit_personnel(
    "人员信息.xlsx",
    "打卡数据导出",
    "人员打卡核查结果.xlsx",
    merge_attendance=True,
)
```
